# VisionSelect 产品可用性评审设计

## 目的

从产品可用性角度评审 VisionSelect，判断它是否适合作为机器视觉工程师、系统集成商和内部研发人员使用的相机、镜头、光源预选型工具。

本次评审要回答的核心问题是：当前软件能否支持快速、可信、可复核的工程预选型，还是存在会降低工程判断可信度的可用性缺口？

本任务只做研究和评估，不修改应用功能。

## 用户假设

主要用户是：

- 懂 FOV、工作距离、像元尺寸、镜头、光源和工业约束的机器视觉工程师或系统集成商。
- 维护型号库、快速生成候选方案的内部研发或应用工程师。

因此，VisionSelect 不需要优先做成面向销售或新手的引导式工具。它更应该优先保证输入效率、推荐可追溯性、数据质量和工程复核能力。

## 外部参考

没有找到和 VisionSelect 完全等价的开源竞品。评审会参考相邻领域的开源工具和公开机器视觉资料，而不是把它们当作直接竞品。

- OpenPnP 视觉工作流：体现真实设备场景中的相机、光源、基准点、Z 高度和标定问题。
- Caliscope 和 Kalibr：体现标定工具对内参、外参、重投影误差和质量反馈的重视。
- Harvester：体现工业相机实际接入时涉及 GenICam、GenTL 和设备节点参数。
- Lensfun：体现镜头特性数据库和畸变校正模型的价值。
- Optiland：作为高阶光学建模参考，覆盖光线追迹、MTF/PSF、畸变、公差和优化。
- 1stVision、Machine Vision Store、Towin、e-con Systems 等公开 FOV/镜头计算器：作为选型输入和理论计算边界的参考。

## 评审维度

### 1. 工程师工作流完整性

检查软件是否覆盖典型工程路径：

- 定义检测需求。
- 估算相机和镜头要求。
- 推荐并对比候选组合。
- 解释推荐理由。
- 展示风险和约束。
- 导出或记录候选方案。
- 维护相机、镜头和光源型号库。

### 2. 推荐可信度

检查推荐结果是否能让有经验的用户理解其工程依据：

- FOV 覆盖关系。
- 像素密度和最小特征/缺陷尺寸关系。
- 工作距离和焦距匹配。
- 镜头像圆与传感器尺寸匹配。
- 畸变和景深风险。
- 相机帧率、接口带宽、位深、动态范围、快门风险。
- 光源有效面积、照明模式、波长/颜色和场景适配。

### 3. 内部工具效率

检查高频用户能否快速完成工作：

- 输入表单是否避免重复操作。
- 表格是否便于扫描和对比。
- 型号库导入导出是否清晰可靠。
- 筛选后导出是否行为可预期。
- 报告是否能作为工程材料复用。
- UI 是否能让用户清楚理解当前型号库和选型状态。

### 4. 错误预防

检查软件是否能减少常见选型错误：

- 单位混乱。
- 像素密度不足。
- 传感器和镜头像圆不匹配。
- 工作距离不可达或容差过小。
- 接口带宽不足。
- 滚动快门或曝光风险不匹配。
- 光源覆盖面积或照明模式不匹配。
- 把理论估算误认为现场验证结果。

### 5. 边界和验证表达

检查软件是否清楚区分：

- 理论预选型。
- 基于型号库的评分。
- 现场验证。
- 相机/镜头标定。
- 最终验收测试。

评审需要指出哪些地方应该加入免责声明、验证清单或可信度标签。

## 评审方案

### 推荐方案：可用性启发式评审

用上面的五个维度审查当前软件，并输出按严重程度排序的问题列表。

优点：最快识别产品问题，适合在不扩大范围的情况下判断下一步优先级。

限制：它能给出产品方向建议，但不能证明算法完全正确。

### 备选方案：端到端工作流走查

选择一个真实选型场景，从输入需求到推荐、对比、报告和型号库维护完整走一遍，并记录每一步的阻力。

优点：证据更贴近真实 UI 和用户操作。

限制：依赖场景选择，可能漏掉更广泛的数据或型号库问题。

### 备选方案：基准矩阵对比

把 VisionSelect 和相邻工具/计算器放在同一张表里，对比输入、计算、风险提示、报告、验证和型号库能力。

优点：差距直观。

限制：容易把非直接竞品的能力误当成必须补齐的功能。

## 推荐输出

本次评审输出一份书面报告，不做代码改动。

报告应包括：

- 总体判断：VisionSelect 是否可作为工程预选型工具使用。
- 优势：当前产品已经做得好的地方。
- 问题：按严重程度和用户影响分组。
- 缺失工作流：当前没有覆盖但对目标用户重要的流程。
- 改进建议：短期、中期、后续再做。
- 非目标：不建议为当前用户群优先投入的能力。
- 证据：必要时引用界面、当前工作流、代码模块和外部参考。

## 非目标

本次评审不会：

- 实现 UI 或算法改动。
- 替代选型算法正确性审查。
- 做完整光学仿真审计。
- 验证真实硬件表现。
- 建设 GenICam 或工业相机采集集成。
- 把软件改造成新手销售向导。

## 成功标准

评审完成后，团队应该能做出清晰产品判断：

- 是否保持当前产品方向。
- 哪些可用性问题需要在扩展功能前先修。
- 哪些工程可信度能力最值得下一步投入。
- 哪些看起来相关但当前不值得做。

## 自检

- 没有包含实现工作。
- 目标用户明确。
- 评审维度贴合机器视觉工程选型。
- 外部参考被定义为相邻基准，而不是直接竞品。
- 输出格式足够具体，后续可以转为研究计划或实施计划。
